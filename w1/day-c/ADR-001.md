# ADR-001: OpenTelemetry SDK vs Vendor-Specific SDKs for Telemetry Collection

## Status
Accepted

## Context
Our platform needs to collect distributed traces, metrics, and logs from 50+ microservices written in Java, Python, and Go. We need to decide whether to use vendor-specific SDKs (e.g., Datadog SDK, New Relic SDK) or a vendor-neutral standard like OpenTelemetry (OTel) SDK. 

Using vendor SDKs provides immediate, seamless integration with their backend platforms, often offering auto-instrumentation that requires zero code changes. However, it tightly couples our application code to a single vendor. 
OpenTelemetry is the CNCF standard for observability. It allows us to instrument once and export to any backend via the OTel Collector.

## Decision
We will use **OpenTelemetry (OTel) SDKs** for instrumenting all microservices and the **OpenTelemetry Collector** as our telemetry pipeline router. We will avoid direct usage of vendor-specific SDKs in application code.

## Consequences
### Positive
* **Vendor Lock-in Avoided:** We can switch from Datadog to an open-source stack (Prometheus/Jaeger/Elasticsearch) or another SaaS vendor (Honeycomb, New Relic) by only changing the OTel Collector configuration, without touching application code.
* **Unified Pipeline:** We can process, filter, and sample telemetry data at the OTel Collector level. This helps control costs (e.g., tail-based sampling for traces).
* **Community Support:** OTel is an industry standard with massive community backing and broad framework support.

### Negative
* **Learning Curve:** Developers and Platform Engineers need to learn OTel concepts (Tracer, Meter, Exporter) instead of relying on familiar, proprietary vendor documentation.
* **Feature Lag:** Some advanced, proprietary features of Datadog APM (e.g., continuous profiler deep integration) might not be fully supported or as plug-and-play via OTel.
* **Maintenance:** We have to manage and host the OTel Collector infrastructure ourselves.

## Alternatives Considered
* **Datadog SDKs:** 
  * *Pros:* Easiest setup, rich proprietary features.
  * *Cons:* Extreme vendor lock-in. Switching costs later would be massive (touching 50+ codebases). Datadog costs can scale aggressively.
* **Direct Push to Kafka from Services:** 
  * *Pros:* Very high throughput.
  * *Cons:* Requires custom libraries in every language to standardize telemetry formats. Reinventing the wheel.
